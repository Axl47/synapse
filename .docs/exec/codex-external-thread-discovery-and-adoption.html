External Codex Thread Discovery and Adoption

Make Synara automatically discover locally persisted Codex threads created by Codex App, the CLI, VS Code, and other app-server clients; place them beside the correct project; and let the user adopt them into Synara without duplicating or silently importing history.

StatusReady for implementation
Created2026-07-12
ScopeContracts, server, persistence query, web sidebar
Primary constraintDiscovery is read-only; adoption is explicit

1. Outcome

After this work:

  • Synara discovers recent, interactive Codex threads for every enabled Codex provider instance.
  • Matched threads appear under From Codex inside the appropriate expanded project.
  • Unmatched threads appear in a compact Other Codex tasks section below Projects.
  • Search includes external threads with an External · Codex badge.
  • Opening an external thread adopts it through the existing import/resume machinery and navigates to it.
  • Already-adopted and Synara-created Codex threads never reappear as external candidates.
  • Opening or reconnecting an adopted thread reconciles persisted messages added by another Codex client.
  • Failures leave the existing Synara project list untouched and offer a retry.

5. Product and architectural decisions

Discovery is automatic; adoption is explicit

Listing an external thread never creates a Synara event, provider session, or local thread. The first click performs adoption. This prevents old history and unrelated accounts from silently flooding Synara.

Identity is provider-instance scoped

The canonical external key is (providerInstanceId, providerThreadId). A raw Codex thread ID is not sufficient because Synara supports multiple Codex homes/accounts. Existing projection session columns are queried for deduplication. Add a non-unique lookup index only if query measurements justify it; do not introduce a second binding table in the first implementation.

Project matching fails closed

Exact project roots, paths contained by one project, and recognized managed worktrees may match automatically. Ambiguous paths, missing paths, deleted directories, and paths outside every project remain unmatched. Adoption revalidates the match server-side rather than trusting a client-provided classification.

Interactive root threads only

The initial release includes non-archived, non-ephemeral root threads from cli, vscode, and appServer. It excludes subagents, compaction/review children, and exec runs. Archived and specialized source filters can be added later without changing the contract shape.

No claim of shared live ownership

This feature synchronizes persisted discovery and history. A different app-server process may own an in-flight turn or approval stream. Synara must show such a candidate as busy when reported by Codex and avoid promising live control until it can safely resume it. True simultaneous control belongs to a later shared-daemon architecture.

6. Contract design

Add provider-neutral orchestration RPCs, while shipping only a Codex implementation initially: 

orchestration.listExternalThreads({ refresh?: boolean })
  -> {
       candidates: ExternalThreadCandidate[];
       refreshedAt: IsoDateTime;
       warnings: ExternalThreadDiscoveryWarning[];
     }

orchestration.adoptExternalThread({
  providerInstanceId,
  externalThreadId,
  projectId
})
  -> { threadId }

ExternalThreadCandidate = {
  provider: "codex";
  providerInstanceId;
  externalThreadId;
  title;
  preview;
  cwd;
  sourceKind;
  status;
  createdAt;
  updatedAt;
  matchedProjectId: ProjectId | null;
  matchKind: "exact" | "descendant" | "managed-worktree" | "unmatched";
}

Keep the candidate intentionally small: thread/list already provides the sidebar metadata, so discovery must not issue thread/read once per row. Return per-instance warnings alongside successful candidates so one broken account does not hide healthy accounts.

7. Execution phases

Phase 1 — Codex thread listing adapter

  1. Add normalized manager types and listExternalThreads() around app-server thread/list.
  2. Send explicit interactive sourceKinds, archived: false, recency sorting, and a bounded page size.
  3. Support pagination internally up to a documented safety cap; return newest candidates first.
  4. Resolve discovery context independently for each enabled Codex provider instance using its configured Codex options.
  5. Normalize protocol drift defensively and reject malformed rows individually rather than failing the whole page.

Primary files: codexAppServerManager.ts, CodexAdapter.ts, ProviderAdapter.ts, and focused tests.

Phase 2 — Discovery service, deduplication, and matching

  1. Create an orchestration discovery handler/service that lists all enabled Codex instances concurrently with a small concurrency bound.
  2. Read existing projection_thread_sessions identities and remove already-bound candidates.
  3. Extract project/worktree classification from the current import route into a shared server module.
  4. Classify every candidate deterministically and attach warnings per provider instance.
  5. Cache normalized results briefly in memory; allow explicit refresh and invalidate on settings/account changes.

Reliability rule: a failure in one account produces a warning for that account while preserving candidates from other accounts.

Phase 3 — Atomic server-owned adoption

  1. Add adoptExternalThread as the authoritative operation used by discovered rows.
  2. Lock by external identity for the duration of adoption and recheck existing bindings before creating anything.
  3. Re-read the external thread, revalidate cwd/project compatibility, then create the Synara thread with the external title and correct Codex instance.
  4. Reuse the import route's provider start, resume cursor, worktree patch, history mapping, and session projection.
  5. If adoption fails after thread creation, perform the same compensating deletion used by manual import and retain the external candidate for retry.
  6. Keep manual ID import working; refactor both paths onto shared server operations rather than maintaining two import implementations.

Race behavior: concurrent adoption requests return the already-adopted Synara thread ID instead of creating duplicates.

Phase 4 — Project-local sidebar and Search UX

  1. Fetch discovery data through a query with stale-time, reconnect refresh, and manual refresh behavior; do not continuously poll while the app is hidden.
  2. Group matched candidates by project without inserting them into persisted thread stores.
  3. Render a conditional From Codex · N disclosure before normal project rows using the shared disclosure motion.
  4. Render Other Codex tasks below Projects only when unmatched candidates exist.
  5. Add external candidates to Search with provider/source badges and project context.
  6. On click, show an inline pending state, call adoption, invalidate discovery, refresh the shell snapshot, and navigate to the returned thread.
  7. For unmatched rows, offer Add project from this folder and Choose project; never guess.

Primary files: Sidebar.tsx, Sidebar.logic.ts, SidebarSearchPalette.tsx, native API wiring, and focused component/logic tests.

Phase 5 — Persisted-history reconciliation

  1. Make Codex snapshot message IDs prefer provider turn/item IDs, with positional IDs retained as a compatibility fallback.
  2. Extract an idempotent history-sync operation that reads thread/read and replays only stable message projections.
  3. Run reconciliation after adoption and before presenting a resumed adopted thread after reconnect/open.
  4. Ensure repeated reconciliation updates existing message IDs rather than appending duplicates.
  5. Keep reasoning/tool/activity reconstruction out of the first slice unless existing runtime ingestion can reuse provider item IDs safely; user and assistant messages are the acceptance boundary.

Phase 6 — Hardening, documentation, and rollout

  1. Add telemetry/logging for discovery duration, candidate count, per-instance failure, adoption, dedupe, and reconciliation.
  2. Document that discovery reflects persisted Codex state, not guaranteed shared live control.
  3. Gate the UI behind a default-on capability only when thread/list succeeds for the installed Codex protocol; otherwise hide the section and retain manual import.
  4. Run focused tests during iteration, then the repository-required final verification pass once explicitly authorized.

8. Detailed edge cases

Case Expected behavior
Synara-created thread appears in Codex listing Suppressed by provider-instance + provider-thread identity.
Same external thread listed twice Normalize to one candidate per external identity; newest metadata wins.
Two accounts expose the same raw UUID Remain distinct because provider instance is part of identity.
Codex thread cwd is a managed worktree Match its owning project and preserve worktree metadata during adoption.
Thread cwd is beneath more than one project root Use the most specific canonical root only when unambiguous; otherwise unmatched.
Thread path no longer exists Show unmatched with disabled adoption and a concise explanation.
External thread is currently active elsewhere Show busy status; do not imply approvals or deltas are attached to Synara.
Candidate disappears between listing and click Adoption fails cleanly, removes pending UI, and prompts refresh.
Adoption request is double-clicked Server identity lock returns one local thread.
One provider instance is misconfigured Other accounts still populate; account-specific warning is non-blocking.
Codex protocol lacks thread/list Capability remains unavailable; manual ID import continues to work.
Externally appended messages after adoption Next open/reconnect reconciles them idempotently.
External rollback rewrites history Do not destructively remove Synara history in v1; log divergence and refresh stable messages only.

9. Verification strategy

Focused automated coverage

  • codexAppServerManager.test.ts: request shape, pagination, malformed rows, source filtering, protocol errors.
  • CodexAdapter tests: normalized external summaries and provider-instance options.
  • Discovery service tests: multi-account partial failure, dedupe, exact/descendant/worktree/unmatched classification.
  • importThreadRoute.test.ts: discovered adoption, race dedupe, compensating deletion, and manual import regression.
  • importedThreadMessages tests: stable provider IDs and repeated reconciliation.
  • Contract encode/decode tests for both new RPCs and warnings.
  • Sidebar.logic.test.ts: grouping, counts, disappearance after adoption, unmatched ordering.
  • Sidebar/Search browser tests: disclosure, badges, pending/error states, keyboard access, and navigation.

Manual acceptance checklist

  • [ ] Create a Codex App task in an existing Synara project; it appears under that project's From Codex section after refresh/reconnect.
  • [ ] Create a CLI task in the same project; source badge and title/preview are correct.
  • [ ] Create a task in an unknown directory; it appears under Other Codex tasks.
  • [ ] Adopt a matched task; it moves once into the normal list and opens with prior messages.
  • [ ] Restart Synara; the adopted task does not return as external.
  • [ ] Add a turn from Codex App, then reopen in Synara; the new persisted messages appear once.
  • [ ] Configure two Codex accounts; each discovers only its own home and one account failure does not blank the other.
  • [ ] Exercise a managed worktree; the task attaches to the parent project and resumes in the original worktree.
  • [ ] Confirm subagents, archived tasks, ephemeral tasks, and exec runs do not clutter the section.
  • [ ] Confirm manual thread-ID import still works.

Per repository instructions, do not run bun fmt, bun lint, or bun typecheck unless explicitly requested in the implementation conversation. Before implementation is considered complete, obtain authorization and run them together as one final heavyweight verification pass. Use bun run test, never bun test.

10. Performance and failure budgets

  • No per-candidate thread/read during listing.
  • Bound pages and concurrent provider-instance discovery.
  • Use Codex state-DB-only listing for routine refreshes when supported; reserve scan-and-repair for first/manual refresh.
  • Cache briefly and refresh on app reconnect, explicit refresh, or stale query focus—not on every sidebar render.
  • A discovery failure must never affect the orchestration shell snapshot or existing thread navigation.
  • Adoption and reconciliation errors are scoped to one external identity.

11. Non-goals

  • Importing every historical or archived Codex thread automatically.
  • Displaying subagent trees as independent external tasks.
  • Mirroring in-flight deltas or approval requests from a separately owned app-server process.
  • Automatically deleting Synara messages after an external rollback.
  • Generalizing discovery to Claude, OpenCode, Kilo, or other providers in this implementation.
  • Replacing manual provider-thread import.
